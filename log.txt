============================================================
1. WHAT THIS IS
============================================================
A static 4-page marketing site, no build step, no framework, no dependencies
except two Google Fonts and (formerly) a CDN script that has since been
removed. Open any .html file directly or serve the folder with any static
file server (e.g. `python3 -m http.server`).

Files:
  index.html      Home — the design "reference" page. Full dawn-to-dusk
                   atmosphere arc (data-atmo 0 -> 1). Never redesigned;
                   every other page derives its shared vocabulary from it.
  about.html       About — warm morning palette, editorial layout.
  services.html    Services — bright midday-into-aqua palette, structured index.
  contact.html     Contact — golden-hour dusk palette, unified form panel.
  styles.css       Single shared stylesheet, ~880 lines. All 4 pages import it.
  site.js          Single shared script, one big IIFE. All 4 pages import it.
  sitemap.xml      Added this session.
  robots.txt       Added this session.
  assets/          Logo/emblem PNG+WEBP pairs, favicons. No photography —
                   the whole site is CSS/SVG-drawn, no raster hero images.

No git repo, no package.json, no CI. Changes are made directly to these files.

============================================================
2. THE CORE IDEA: ONE ATMOSPHERE, FOUR WINDOWS
============================================================
Every page shares a fixed-position `.atmosphere` layer (z-index 0) sitting
behind the actual content (z-index 2): a sky gradient (#lp-sky), sun/rays,
mist layers, drifting particles, a hills SVG, a water-fill div, a
grove/landscape SVG, a vignette and film grain.

site.js drives all of it from ONE continuous "dawn to dusk" color arc — a
`stops` array of 7 keyframes (t=0 to t=1) blended with linear interpolation
based on scroll position. Each page occupies a slice of that arc via
`data-atmo-start` / `data-atmo-end` attributes on <body>:

  index.html     0.00 -> 1.00   (the full journey — dawn to dusk)
  about.html     0.12 -> 0.40   (dawn edge into warm morning green)
  services.html  0.42 -> 0.72   (pale midday mint into aqua/water)
  contact.html   0.84 -> 1.00   (deep dusk green into gold)

This means navigating between pages feels like moving through one continuous
world at a different time of day, not four disconnected templates. Do not
change these ranges without checking what visually sits at each page's start
and end (see the `stops` array in site.js for the actual color keyframes).

The reveal system: any element with `data-lp-float` fades/blurs based on its
distance from the vertical center of the viewport (opacity/blur computed each
scroll frame). This is why content "breathes" in as you scroll. It's also why
a scroll-snap mechanism was tempting (see section 5) — the reveal system's
own math already defines "the point of maximum clarity" as viewport-center.

============================================================

       + Jost (body, heavy letter-spacing, uppercase for labels).

  --aqua/--aqua-deep/--aqua-panel                                Services' accent family (teal/turquoise)
  --dusk-gold/--dusk-deep/--dusk-panel                           Contact's accent family
  --ease-out / --ease-inout / --ease-spring                      motion easing tokens

Per-page identity (this is the answer to "why does each page look different"):
  About     — moss/gold, meadow hills with a scalloped orchard-canopy texture
              baked into the hill silhouette itself (not the separate grove
              layer — see section 6 for why). Asymmetric editorial hero
              (name left, medallion right). Vertical "journey" timeline
              instead of a card grid. Pull-quote breaks up the story text.
              Framed keepsake CTA box with gold corner brackets.
  Services  — aqua/teal, wide smooth sweeping dune hills (NOT the original
              angular mesa version — see section 6, it was redesigned after
              feedback that mesas felt too angular/lifeless). Turquoise
              water-fill override (`body.pg-services #lp-water .water-fill`).
              Structured index-list of offerings (numbered rows with anchor
              ids svc-coaching/svc-reiki/svc-transformation/svc-discovery),
              not a card grid. TOC pills in the hero jump to each row.
              Connected-line process stepper and benefit strip.
  Contact   — gold/dusk, one unified `.ct-frame` panel (gold corner brackets,
              internal divider) instead of two separate floating cards.
              "What happens next" reassurance strip below the form.
  Home      — untouched reference. Centered hero, list-style services
              preview, 3-step path, stillness quote + testimonial card.

Shared connective tissue (do not fork these per-page — they're what makes
the site read as ONE site): nav/footer markup, .cta-pill button style,
eyebrow + roman-numeral section labels (I · Presence, II · The Work,
III · Begin Your Journey), the data-lp-float reveal motion, page-in fade-up.

============================================================
4. RESPONSIVE / A11Y NOTES
============================================================
Breakpoints in use: 1500 (large desktop), 1024 (tablet landscape), 900 (nav
switches to hamburger), 768 (tablet portrait), 600 (large phones), 400 (small
phones), plus a max-height:560px landscape-phone catch. Follow these existing
breakpoints rather than inventing new ones.

Fixed an accessibility bug this project: the hero `<section aria-labelledby>`
on about/services/contact referenced an id that didn't exist on the h1. Now
every hero h1 has a matching id (about-hero-title / svc-hero-title /
contact-hero-title).

`prefers-reduced-motion: reduce` is respected throughout — disables particles,
loader, float reveals, and (per section 7) the magnetic button JS.

--- Session B: landscapes, scroll magnet v1, magnetic hover ---
User: home/contact are good, but about/services need more contrasting
landscapes than "just rolling hills", a scroll-snap "magnet" so content
doesn't rest mid-blur, and better button hover states.

    band math in site.js's `update()` against the page's data-atmo range
    before assuming a layer will render.

    snapped to nearest `#main > .zone, #main > .section` top edge (offset by

    .cta-pill/.nav-cta/.form-submit/.svc-toc a/.contact-arrow, translates
    the button toward the cursor (clamped ±10/±8px) + a spring-eased
    snap-back on mouseleave. Gated behind
    `matchMedia('(hover:hover) and (pointer:fine)')` and !reduceMotion so
    touch devices just get the plain CSS :hover. Added a diagonal shine-sweep
    pseudo-element to the primary CTA buttons. Added --ease-spring token
    (cubic-bezier(0.34,1.56,0.64,1), a slight overshoot/bounce).

--- Session H: mobile scroll jank + contact form fade fix, first git push ---
User reported a to-do list: mobile scrolling glitchy at the start of home
(near the sun) and on about, and the mobile contact form blurs/fades before
you can reach the submit button. Also: this session set up git for the
first time and pushed to https://github.com/norvodesigns/thegreatlights.git.

  - Root-caused the scroll jank to two things in site.js's shared `update()`
    function (runs on every scroll frame, every page):
      1. The sun element was animating `width`/`height`/`top` every frame —
         all three are layout-triggering CSS properties, so every scroll
         tick forced a synchronous reflow before the browser could paint.
         Fixed by splitting #lp-sun into a fixed-size (600px, matching the
         largest `sz` value in the stops table so scale never exceeds 1)
         outer element that JS only ever moves/scales via `transform`
         (translate + scale, compositor-only, no layout cost) plus opacity,
         and an inner `.sun-glow` child that owns the radial-gradient
         background and the existing `lp-breathe` pulse keyframe (kept as
         a separate element specifically so JS's inline transform and the
         CSS keyframe's transform don't fight over the same property on the
         same element).
      2. The `[data-lp-float]` reveal loop called `getBoundingClientRect()`
         on every single floating element on every scroll frame — expensive
         on its own, and guaranteed to force a synchronous layout recompute
         immediately after the sun's old width/height/top writes. Fixed by
         caching each float's document-relative top/height once
         (`measureFloats()`), re-measuring only on `document.fonts.ready`,
         `window.load`, and a debounced `resize`/`visualViewport resize` —
         not on every scroll frame. The per-frame loop is now pure
         arithmetic (`cachedTop - scrollY`) plus opacity/transform/filter
         writes, all compositor/paint-only.
    Both changes are real, verifiable wins (eliminated the only two
    layout-triggering property writes in the whole scroll path) — but note
    this was reasoned from code inspection, not a live mobile device
    profile (not available in this environment). If it's still glitchy on
    an actual phone after this, the next place to look is the 46 particle
    divs' concurrent CSS animations in #lp-particles, or the backdrop-filter
    blur cost on the frosted buttons/nav — both plausible but unconfirmed.
  - Contact form fix: `.ct-frame` (the whole unified form panel added back
    in the per-page-redesign session) had `data-lp-float` on it, making the
    ENTIRE tall panel — aside text, all fields, and the submit button — one
    single reveal unit. The reveal formula fades/blurs based on distance
    from the element's own centre to the viewport centre; for a block
    taller than about 1.3x the viewport (easily the case for this form on a
    short mobile screen, once you stack name/email/phone/interest/message/
    submit), by the time you've scrolled down far enough to see the submit
    button the block's centre is far enough away that it's already fading
    out — exactly backwards from what you want while actively filling out a
    form. Fix: removed `data-lp-float` from `.ct-frame` entirely, so the
    form is always fully opaque and unblurred regardless of scroll
    position. Left `.ct-next` (the short "what happens next" strip below
    the form) with its reveal intact — it's non-interactive and not what
    was reported. Verified via computed style after scrolling the submit
    button into view: opacity 1, filter none.
  - This was also the first time this project touched git. No repo existed
    before this session. Initialized one, added a .gitignore for
    .DS_Store, and pushed to the (previously empty) GitHub remote. See the
    note in section 6 about how the push was authenticated — don't reuse
    that pattern for future pushes without a credential helper.

--- Session I: focus window, shadows, mobile-menu ring bug, scroll smoothing ---
User to-do list: the reveal "in focus" window felt too narrow; text needed
more drop shadow for legibility; the mobile menu's Home link sometimes got
a stray gold outline rectangle "seemingly randomly"; and scenery transitions
still felt "framy, almost like stop motion" on mobile.

  - Focus window: the opacity/blur formula in the [data-lp-float] loop
    (site.js `update()`) used `1.28 - |d|*1.7` where d is distance from
    viewport centre in viewport-heights — that only reached opacity 1
    within about ±16.5% of centre (~33% of the screen), fading everywhere
    else. Replaced with a band-based formula using the existing `band()`
    helper: `band(frac,0,0.1) * (1-band(frac,0.9,1))` where frac = element
    centre / viewport height. This gives exactly opacity 1 across the
    middle 80% of the viewport (10% fade band at the very top, 10% at the
    very bottom) — verified by evaluating the formula directly for
    frac = 0..1 in the browser. The translateY parallax lift was left as-is
    (still keyed off the old `d` distance metric) — only opacity/blur
    changed, since the parallax motion wasn't part of the complaint.
  - Text shadow: strengthened the existing dark text-shadow (used on
    .zone-home/.zone-path/.zone-stillness/.zone-contact/.hero-cream — the
    light-text-on-raw-atmosphere zones) from a 2-layer
    `0 2px 20px rgba(...,.22), 0 1px 4px rgba(...,.16)` to a 3-layer
    tight+medium+wide combo at higher opacity. Also added `.tone-cream` to
    that same group (it wasn't covered before, and is used on contact.html's
    form section wrapper). NEW this session: added a subtle light/white
    halo shadow (`0 1px 1px rgba(255,255,255,.5), 0 2px 16px
    rgba(255,255,255,.4)`) to `.tone-ink`, `.tone-ink-green`, and
    `.hero-ink` — the shared dark-ink utility classes used throughout
    About/Services — so dark text also gets a touch of separation from the
    atmosphere instead of only light text being covered.
  - Mobile menu focus-ring bug: root cause was `f[0].focus()` in
    `openMenu()` (site.js) — moving focus into the opened menu for
    keyboard/screen-reader users, which is correct a11y practice, but some
    browsers show the `:focus-visible` gold ring for script-triggered focus
    inconsistently, which is exactly what read as "seemingly random" to the
    user. First attempt: added a `.no-ring` class before the focus() call,
    removed via `setTimeout(...,400)`. WRONG — verified via a page-clock
    -precise test (sampling activeElement/classList/:focus-visible at fixed
    delays via the page's own setTimeout, not Python-side sleep, which
    proved unreliable for this) that the ring would pop back in ~460ms
    after opening even if the user hadn't done anything — i.e. the same bug,
    just on a delay. Fixed properly: remove `.no-ring` on the element's own
    `blur` event instead of a timer, so the ring stays suppressed for
    exactly as long as that specific programmatic focus holds (however
    long that is), and a genuine subsequent Tab keypress — which moves
    focus to a *different* element that never got `.no-ring` — still shows
    the ring normally. Verified both cases explicitly: focus held 2 full
    seconds with no ring appearing, and a real Tab keypress afterward
    showing the ring on "Services". Applied the same `focusQuiet()` helper
    to `closeMenu()`'s restore-focus call too, for consistency.
  - Scroll "stop motion" smoothing: added a short `transition: ... .12s
    linear` to every atmosphere element whose background/opacity/transform
    site.js sets on scroll (#lp-sky, #lp-rays, #lp-sun, #lp-mist-a/b,
    #lp-hills, #lp-water, #lp-grove) and to `[data-lp-float]`'s
    opacity/transform/filter. The theory: mobile browsers throttle native
    'scroll' event dispatch during an active touch-drag more than during
    momentum scroll or desktop wheel scroll, so JS was sometimes only
    getting to update these styles in sparse bursts — without a CSS
    transition, each update hard-snaps to its new value, which reads as
    choppy/stepped ("stop motion") when updates are infrequent. The
    transition lets the compositor interpolate smoothly between whatever
    values JS manages to set, independent of how often it actually runs.
    120ms is short enough to still feel tightly coupled to scroll input.
    This is reasoned from documented mobile browser behaviour, not measured
    on a real device in this environment — if it's still choppy after this,
    it's likely the 'scroll' event throttling itself rather than anything
    CSS can paper over, and the next step would be also listening to
    `touchmove` directly to sample position more eagerly during active
    drags.
  - Lesson for future debugging sessions: when timing a JS effect via
    browser automation, don't trust `asyncio.sleep(N)` on the Python side
    to correspond to N real milliseconds of page time — CDP round-trip
    overhead accumulates and produced at least one false negative this
    session (a test that looked like the fix wasn't working, when it
    actually was — see the focus-ring debugging above). Schedule the
    check itself inside the page via `setTimeout`/`performance.now()` and
    return the result, instead of polling from outside with assumed delays.

6. THINGS TO KNOW BEFORE YOU TOUCH THIS CODE
============================================================
- No Lenis. Don't re-add a smooth-scroll library without asking — it was
  explicitly removed twice-over (once for sluggishness, its replacement
  scroll-magnet was then also removed by explicit request). Native scroll
  is the current, confirmed-preferred state.
- No scroll-snap / magnet mechanism. Was built, iterated twice, then
  explicitly removed. Don't re-add without the user asking for it again.
- The #lp-grove layer is invisible on About (data-atmo range never reaches
  its 0.66-0.86 visibility band) and only partially visible near the very
  end of Services (ramps 0->~0.3 opacity from 0.66 to Services' max of
  0.72). Contact's grove IS fully visible (range 0.84-1.0 sits inside the
  band) and still uses the ORIGINAL pine-forest art, unchanged. Home's
  grove is also the original, fully visible near the end of its range.
- About/Services/Contact only ever use dark ink text on light-toned
  sections — this was a deliberate structural choice (not an accident) and
  is *why* those pages can't have the "white text on bright background"
  problem that the home page's raw-atmosphere cream-text zones can.
- .svc-toc / .cta-pill / .nav-cta etc. now have backdrop-filter blur at
  REST, not just on :hover. If you add a new glassy button, give it a
  visible background+blur at rest too — several of the ones fixed this
  session had literally no background until hovered, which is why they
  were hard to see against bright parts of the atmosphere.
- Git repo exists as of Session H, pushed to
  https://github.com/norvodesigns/thegreatlights.git, branch `main`, with
  upstream tracking already set up — a plain `git push` works. The stored
  remote URL in .git/config has no credentials in it (by design). When a
  token/credential is needed for a push, don't pass it inline in the `git
  push <url>` command — it lands in shell history and the process list.
  Use a credential helper, an env var consumed by a GIT_ASKPASS script, or
  have the user authenticate interactively instead.
- Contact form posts via `action="mailto:..."` — this opens the visitor's
  own email client rather than actually submitting anywhere server-side.
  Works, but fragile (fails silently if they have no mail client
  configured) — see the checklist for a recommended fix.
- The float-reveal system (data-lp-float) caches element positions on load
  instead of measuring every scroll frame (see Session H). If you add new
  floating content dynamically after page load (via JS, not present in the
  initial HTML), it won't be in the cache — call measureFloats() again, or
  add it before the cache is first built.
- Don't put data-lp-float on anything containing form fields or other
  interactive controls unless it's short (well under one viewport tall).
  The reveal formula fades based on distance from the element's own centre,
  which actively fights usability on tall interactive blocks — see the
  Session H contact-form fix for why.
